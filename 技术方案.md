# Subly 省心订阅 — 整体技术方案

> 目标：自托管的订阅 / 续费 / 保号管理系统，内置 SQLite 零配置，支持多用户、多货币、多语言、多主题，并通过 Telegram + Bark 双通道提醒用户及时续费。

---

## 1. 总体架构

```
┌─────────────────────────────────────────────────────────┐
│                     浏览器 (Web UI)                       │
│   Vue 3 · 多语言(中/英/俄) · 多主题 · 仪表盘 · 日历 · 报表  │
└───────────────────────────┬─────────────────────────────┘
                            │ HTTPS / REST API
┌───────────────────────────▼─────────────────────────────┐
│                      后端服务 (FastAPI)                   │
│  鉴权(JWT) · 多用户 · 订阅 CRUD · 报表统计 · 货币换算       │
├──────────────┬───────────────┬──────────────┬────────────┤
│  定时任务     │  通知服务      │  汇率拉取     │  图标服务   │
│ (到期扫描)    │ TG + Bark     │ (汇率 API)   │ 上传/缓存   │
└──────┬───────┴───────┬───────┴──────┬───────┴────────────┘
       │               │              │
   ┌───▼───┐      ┌────▼────┐    ┌────▼─────┐
   │SQLite │      │外部通知  │    │汇率 API  │
   │/app/data│    │TG/Bark   │    │第三方    │
   └───────┘      └─────────┘    └──────────┘
```

部署：Docker / Docker Compose，一体化镜像包含后端 API 与前端静态文件。持久化目录为 `/app/data`，保存 SQLite 数据库、上传图标和内置图标库缓存。

## 2. 技术选型

| 层 | 当前选型 | 说明 |
|----|----------|------|
| 后端 | **Python + FastAPI** | REST API、依赖注入、后台任务清晰 |
| ORM / 数据库 | **SQLAlchemy + SQLite** | 零配置，数据库文件持久化在 `/app/data/subly.db` |
| 迁移 | 轻量在线迁移脚本 `migrate.py` | 启动时按列补齐旧库结构，适合单机自托管 |
| 前端 | **Vue 3 + Vite + Pinia** | 轻量、响应快，适合自托管 Web UI |
| 国际化 | vue-i18n | 中文 / English / Русский |
| 定时任务 | APScheduler | 每日扫描到期订阅并发送提醒 |
| 通知 | Telegram Bot API + Bark API | 两个通道可同时启用、独立记录日志 |
| 部署 | Docker 多阶段构建 + 可选 Caddy | 支持 amd64 / arm64 与 NAS 部署 |

## 3. 外部 API 与网络访问

### 3.1 Telegram Bot API

- **创建 Bot**：通过 @BotFather 获取 `bot_token`。
- **获取 chat_id**：用户需先和机器人对话，再通过网页设置获取或填写。
- 关键接口：
  - `getMe`：验证 Bot Token。
  - `sendMessage`：发送续费提醒。
  - `getUpdates`：辅助获取 Chat ID。
- 支持 Telegram API 反代和 HTTP 代理，适配不同网络环境。

### 3.2 Bark API

- 用户在 iOS Bark App 中复制 Device Key，填写到网页「设置 → Bark 配置」。
- 支持自建 Bark 服务器地址。
- 支持可选 TTL：
  - 留空：不传 `ttl` 字段，使用 Bark 默认行为。
  - `0` 或其他非负整数：测试推送与定时提醒都会显式透传给 Bark。
  - 前端与后端都拒绝负数、小数、指数形式等非非负整数输入。

### 3.3 汇率 API

- 默认使用免费汇率源（如 `open.er-api.com`）。
- 后端定时拉取并缓存汇率，订阅金额统一换算到用户的基准货币。
- 支持自定义货币与手动汇率覆盖。

### 3.4 图标 favicon 下载

内置图标库不预置所有图片文件，而是按需下载并缓存服务 favicon：

1. 优先访问目标站点 `https://<domain>/favicon.ico`。
2. 解析站点首页的 `rel=icon` / `apple-touch-icon` 等链接。
3. 尝试公共 favicon provider。
4. 可选尝试 Google favicon provider。
5. 全部失败时返回后端生成的可见 SVG fallback（稳定颜色 + 首字母），避免透明空白图标。

安全与稳定性约束：

- 只缓存 PNG / ICO / WEBP / JPEG，不缓存远端 SVG。
- 限制单次下载超时与最大字节数。
- 保留单 slug 失败缓存、全局熔断和并发信号量，避免冷缓存时的请求风暴。
- 日志只记录事件、slug、provider 和错误类型，不记录完整外部 URL、query、headers、token 或 API key。

相关环境变量：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `ICON_FETCH_ENABLED` | `true` | 是否允许内置图标库联网下载 favicon |
| `ICON_FETCH_GOOGLE_ENABLED` | `true` | 是否启用 Google favicon provider |
| `ICON_FETCH_TIMEOUT_S` | `2.0` | 单次图标下载超时秒数 |
| `ICON_FETCH_MAX_BYTES` | `262144` | 单个图标最大下载字节数 |

## 4. 数据模型（核心表）

```
users            (id, username, email, password_hash, role,
                  locale, theme, base_currency,
                  telegram_enabled, telegram_bot_token, telegram_chat_id,
                  bark_enabled, bark_device_key, bark_server,
                  bark_sound, bark_group, bark_ttl,
                  created_at)

categories       (id, user_id?, name, icon, is_system, sort)
                 # 系统预置 + 用户自定义

payment_methods  (id, user_id?, name, icon, is_system)
                 # 信用卡 / Apple Pay / 支付宝 / 微信 / PayPal / 银行卡等

currencies       (code, name, symbol, is_custom, user_id?)
exchange_rates   (base, quote, rate, updated_at)

subscriptions    (id, user_id, name, icon, category_id, payment_method_id,
                  amount, currency, billing_type,       # recurring | one_time
                  cycle, cycle_count,                   # day/week/month/year + N
                  start_date, next_renewal_date, end_date?,
                  is_active, auto_renew, url, notes)

reminders        (id, subscription_id, days_before, channel)
notification_log (id, subscription_id, channel, sent_at, status, message)
```

要点：

- `billing_type` 区分周期订阅与一次性买断。
- `next_renewal_date` 由周期推算，续费后按规则滚动到下一周期。
- Bark TTL 是用户级设置，允许 `NULL`，旧数据通过迁移脚本自动补列。
- 图标路径仍保存为字符串，可是 Emoji、上传图片 URL、外部 URL 或 `/api/icons/library/{slug}.png`。

## 5. 核心功能模块

1. **鉴权与多用户**：注册 / 登录、JWT、管理员 / 普通用户角色、用户数据隔离。
2. **订阅管理**：周期 / 买断、自定义图标、分类、付款方式、续费规则。
3. **仪表盘**：当月支出、年度支出、即将到期、最近订阅、分类总览。
4. **分类与付款方式**：系统预置 + 用户自定义，支持排序与图标。
5. **多货币**：全球常用货币、自定义货币、实时汇率、按用户基准货币统计。
6. **多语言 / 多主题**：中文 / English / Русский，5 套主题。
7. **日历视图**：按到期日渲染订阅事件。
8. **报表分析**：支出洞察、分类排行、永久购买、即将续费、已过期订阅。
9. **通知提醒**：每日定时扫描 → 命中提醒规则 → Telegram / Bark 独立发送 → 写入日志。
10. **图标服务**：Emoji、上传图片、URL 导入、内置图标库 favicon 下载与可见 fallback。
11. **备份恢复**：单用户 JSON 备份；管理员整站备份 / 恢复全部成员数据。

## 6. 提醒调度逻辑

```
每天 REMINDER_SCAN_TIME 定时任务:
  for 每个 active 周期订阅:
     for 每条 reminder 规则 (提前 N 天):
        if next_renewal_date - today == N 且 今天未发过:
           组装提醒文案
           if 用户启用 Telegram: 发送 Telegram 并记录 notification_log(channel="telegram")
           if 用户启用 Bark: 发送 Bark（包含可选 bark_ttl）并记录 notification_log(channel="bark")
  续费完成后: next_renewal_date 按订阅规则滚动
```

两个通知通道互不依赖：一个失败不会阻断另一个发送。

## 7. 图标显示逻辑

前端统一使用 `ServiceIcon` 渲染订阅图标：

- `src` 是图片 URL 时优先显示图片。
- 图片加载失败时切到 fallback。
- 旧版本透明 1x1 占位图会通过 `naturalWidth <= 1 && naturalHeight <= 1` 识别并切到 fallback。
- fallback 优先使用 Emoji，否则按服务名 / 订阅名首字符生成稳定颜色的字母图标。

后端图标库接口仍保持 `/api/icons/library/{slug}.png` 路径兼容，但响应的真实 `Content-Type` 可以是 PNG / ICO / WEBP / JPEG / SVG fallback。

## 8. 部署方案

### 8.1 Docker 运行

```bash
docker run -d --name subly \
  -p 8842:8000 \
  -e JWT_SECRET="$(openssl rand -hex 32)" \
  -e ADMIN_USERNAME=admin \
  -e ADMIN_PASSWORD=admin123 \
  -e ADMIN_EMAIL=admin@example.com \
  -e TZ=Asia/Shanghai \
  -v subly_data:/app/data \
  --restart unless-stopped \
  yourname/subly:latest
```

### 8.2 Docker Compose

```bash
docker compose -f docker-compose.hub.yml up -d
```

### 8.3 镜像发布

GitHub Actions `docker-publish.yml` 支持：

- 推送到 `main` 或 `v*` tag 时自动构建并发布。
- 手动执行 `workflow_dispatch` 时填写 Docker 镜像版本号（如 `1.0.0`）。
- 同时发布 Docker Hub 与 GHCR 的 `latest` 和指定版本 tag。
- 版本号会按 Docker tag 规则校验。

## 9. 安全与隐私约束

- `JWT_SECRET` 必须使用随机强密钥。
- 管理员初始密码首次登录后应立即修改。
- 日志不得记录 JWT、password、token、API key、request body、headers、query、完整外部 URL、完整订阅名、notes、remark、family_members 或 IP 地址。
- 图标下载错误只记录必要的事件和错误类型，不泄露用户输入的完整 URL。
- 备份文件包含个人订阅数据，导出后应妥善保管。
