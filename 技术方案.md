# Subly — 整体技术方案

> 定位：你的自托管续费雷达，集中管理订阅、域名、VPS、保号套餐与提醒通道；内置 SQLite 零配置，中文单语言，支持多用户、注册审核、多货币、多主题、Telegram + Bark 双通道提醒、内置服务管理、备份恢复与实时日志。

---

## 1. 总体架构

```text
┌────────────────────────────────────────────────────────────┐
│                         浏览器 Web UI                       │
│ Vue 3 · Vite · Pinia · Vue Router · vue-i18n（仅中文）       │
│ 雷达工作台 · 订阅账本 · 续费日历 · 财务报表 · 管理控制台       │
└──────────────────────────────┬─────────────────────────────┘
                               │ HTTPS / REST API
┌──────────────────────────────▼─────────────────────────────┐
│                        FastAPI 后端                         │
│ JWT 鉴权 · 用户审核 · 订阅 CRUD · 报表统计 · 备份恢复         │
│ 设置管理 · 通知日志 · 实时活动日志 · 静态前端托管             │
├──────────────┬────────────────┬────────────────┬───────────┤
│ APScheduler  │ 通知服务        │ 汇率服务        │ 图标服务  │
│ 到期扫描      │ Telegram+Bark   │ 汇率拉取/缓存   │ 上传/缓存 │
└──────┬───────┴────────┬───────┴────────┬───────┴───────────┘
       │                │                │
┌──────▼──────┐   ┌─────▼─────┐    ┌─────▼─────┐
│ SQLite       │   │ TG / Bark │    │ 汇率 API   │
│ /app/data    │   │ 外部推送  │    │ 第三方数据 │
└──────────────┘   └───────────┘    └───────────┘
```

部署形态：

- 单镜像同时托管 FastAPI API、上传图标静态资源与前端 SPA。
- Dockerfile 使用 `node:20-alpine` 构建前端，再用 `python:3.12-slim` 运行后端。
- 容器内服务监听 `0.0.0.0:8000`，NAS / Docker Hub compose 默认映射到宿主机 `8842`。
- 源码部署的 `docker-compose.yml` 额外使用 Caddy，在 `80/443` 反代到 `app:8000`。
- 持久化目录为 `/app/data`，保存 SQLite 数据库、上传图标和内置图标库缓存。

## 2. 技术选型

| 层 | 当前选型 | 说明 |
|----|----------|------|
| 后端 | Python 3.12 + FastAPI | REST API、依赖注入、后台任务清晰 |
| ORM / 数据库 | SQLAlchemy + SQLite | 零配置，数据库文件持久化在 `/app/data/subly.db` |
| 配置 | pydantic-settings | 通过 `.env` / 环境变量读取部署配置，忽略未知变量 |
| 迁移 | 轻量在线迁移脚本 `migrate.py` | 启动时按列补齐旧库结构，适合单机自托管 |
| 调度 | APScheduler | 每日按 `REMINDER_SCAN_TIME` 扫描到期订阅 |
| 通知 | Telegram Bot API + Bark API | 两个通道可同时启用、独立发送、独立记录日志 |
| 前端 | Vue 3 + Vite + Pinia + Vue Router | 单页应用，构建后由后端静态托管 |
| 文案 | vue-i18n（仅中文 `zh`） | 保留集中式文案管理，不再维护英文 / 俄文语言包 |
| 部署 | Docker 多阶段构建 + 可选 Caddy | 支持 NAS、普通 Docker、源码 compose 与镜像 compose |

## 3. 配置与环境变量

后端配置源为 `backend/app/config.py` 中的 `Settings`。关键变量：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DB_PATH` | `data/subly.db` | SQLite 数据库文件路径；容器内建议保持默认并持久化 `/app/data` |
| `JWT_SECRET` | `change-me` | 登录令牌密钥，生产环境必须改成随机强密钥 |
| `JWT_ALGORITHM` | `HS256` | JWT 算法，一般不需要修改 |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | `60` | Access Token 有效期 |
| `REFRESH_TOKEN_EXPIRE_DAYS` | `14` | Refresh Token 有效期 |
| `TZ` | `Asia/Shanghai` | 应用时区 |
| `LOG_LEVEL` | `INFO` | 后端 stdout 日志级别 |
| `SLOW_REQUEST_MS` | `1000` | 超过该毫秒数的请求额外记录 `slow_request` |
| `APP_PUBLIC_URL` | 空 | 对外可访问地址（如 `https://subly.example.com`）；仅影响 Bark 测试推送的点击跳转，真实续费提醒点击地址取订阅 `url` |
| `REMINDER_SCAN_TIME` | `09:00` | 每天扫描到期订阅并发送提醒的时间 |
| `REQUIRE_ADMIN_APPROVAL` | `true` | 新用户注册后是否需要管理员审核 |
| `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASSWORD` / `SMTP_FROM` / `SMTP_TLS` | 空 / `587` / `true` | SMTP 邮箱验证码配置 |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` / `ADMIN_EMAIL` | `admin` / `admin123` / `admin@example.com` | 管理员账号；启动时若该用户名不存在则创建 |
| `TELEGRAM_BOT_TOKEN` | 空 | 仅声明保留，当前不参与发送或验证；Telegram Bot Token 以用户级网页设置为准 |
| `EXCHANGE_API_BASE` / `EXCHANGE_API_URL` / `EXCHANGE_API_KEY` | `USD` / `https://open.er-api.com/v6/latest/` / 空 | 汇率数据源配置 |
| `ICON_FETCH_ENABLED` | `true` | 是否允许内置图标库联网下载 favicon |
| `ICON_FETCH_GOOGLE_ENABLED` | `true` | 是否启用 Google favicon provider |
| `ICON_FETCH_TIMEOUT_S` | `2.0` | 单次图标下载超时秒数 |
| `ICON_FETCH_MAX_BYTES` | `262144` | 单个图标最大下载字节数 |
| `ICON_FETCH_CONCURRENCY` | `6` | 冷缓存时最多并发下载 favicon 数量 |
| `ICON_FETCH_SVG_ENABLED` | `true` | 是否接受并消毒缓存远端 SVG favicon |

Bark 的 Device Key、服务器、提示音、分组与 TTL 均在网页「设置」中按用户配置，无对应环境变量。`APP_PUBLIC_URL` 环境变量仅影响 Bark **测试推送**的点击跳转地址（真实续费提醒的点击地址取订阅自身 `url`）。

## 4. 外部 API 与网络访问

### 4.1 Telegram Bot API

- 通过 `@BotFather` 创建 Bot 并获取 Bot Token。
- 用户需先与机器人对话，再在网页设置中获取或填写 Chat ID。
- 关键接口：
  - `getMe`：验证 Bot Token。
  - `getUpdates`：辅助获取 Chat ID。
  - `sendMessage`：发送续费提醒。
- 用户级设置支持 Telegram API 反代与 HTTP 代理，适配不同网络环境。

### 4.2 Bark API

- 用户在 iOS Bark App 中复制 Device Key，填写到网页「设置 → Bark 配置」。
- 支持自建 Bark 服务器地址。
- 支持提示音、分组与可选 TTL：
  - 留空：不传 `ttl` 字段，使用 Bark 默认行为。
  - `0` 或其他非负整数：测试推送与定时提醒都会显式透传给 Bark。
  - 前端与后端都拒绝负数、小数、指数形式等非非负整数输入。

### 4.3 汇率 API

- 默认使用 `https://open.er-api.com/v6/latest/`。
- 支持可选 `EXCHANGE_API_KEY`。
- 后端缓存汇率，订阅金额统一换算到用户的基准货币。
- 缺汇率时 `convert()` 不报错，仅记录一次 warning 并返回原金额，避免汇率缺失阻断展示。
- 刷新触发：应用时区每天 `04:00` 自动刷新；缓存超过 12 小时且打开相关页面时自动刷新；管理员可手动调用 `/api/currencies/rates/refresh`。
- 用户可维护自定义货币与自定义显示符号。

### 4.4 favicon 下载与缓存

内置服务库的图标按需下载并缓存：

1. 优先访问目标站点 `https://<domain>/favicon.ico`。
2. 解析站点首页的 `rel=icon` / `apple-touch-icon` 等链接。
3. 依次尝试公共 favicon provider：unavatar、iconhorse、simple-icons、jsDelivr simple-icons、DuckDuckGo。
4. 可选尝试 Google favicon provider（`ICON_FETCH_GOOGLE_ENABLED`）。
5. 全部失败时返回后端生成的可见 SVG fallback（稳定颜色 + 首字母），避免透明空白图标。

安全与稳定性约束：

- 允许缓存 PNG / ICO / WEBP / JPEG。
- 允许缓存远端 SVG，但必须先消毒：移除脚本、事件属性、外链、`data:` URL 等风险内容；可用 `ICON_FETCH_SVG_ENABLED=false` 禁用。
- 限制单次下载超时、最大字节数与冷缓存并发数。
- 保留单 slug 失败缓存、全局熔断和并发信号量，避免冷缓存请求风暴。
- 失败时提供可见 fallback；限流 fallback 使用 `no-store`，避免浏览器长期缓存临时失败结果。

## 5. 数据模型（核心表）

以下以 `backend/app/models.py` 为准，只列核心字段与用途。

```text
users
  id, username, email, password_hash
  is_admin, is_active
  email_verified, is_approved, email_code, email_code_expires
  locale, theme, base_currency, category_order
  telegram_enabled, telegram_bot_token, telegram_chat_id,
  telegram_admin_id, telegram_api_base, telegram_proxy
  bark_enabled, bark_device_key, bark_server, bark_sound,
  bark_group, bark_ttl
  created_at

categories
  id, user_id?, name, icon, color, is_system, sort

payment_methods
  id, user_id?, name, icon, is_system

icon_library_services
  id, name, domain, website
  category, category_keys
  slug, is_active, sort, source
  created_at, updated_at

currencies
  code, name, symbol, is_custom, user_id?

exchange_rates
  id, base, quote, rate, updated_at

subscriptions
  id, user_id
  name, plan, icon, url, notes, remark, ipv4, ipv6
  category_id, payment_method_id, bundle_id
  amount, currency
  billing_type, is_keepalive, cycle, cycle_count
  start_date, next_renewal_date, end_date, last_renewed_at
  is_active, auto_renew, show_in_calendar, sort
  family_members
  remind_days_before
  created_at

bundles
  id, user_id, name, note, created_at

activity_log
  id, user_id?, username?, level, action, detail, created_at

notification_log
  id, subscription_id, user_id, days_before, channel,
  status, message, sent_at
```

要点：

- `users.locale` 是历史兼容字段，当前界面固定中文，不再暴露给前端用户设置。
- 用户审核由 `email_verified` 与 `is_approved` 表达；是否需要审核由 `REQUIRE_ADMIN_APPROVAL` 控制。
- Telegram / Bark 是用户级设置，可按账号独立启用。
- `categories`、`payment_methods` 同时支持系统预置与用户自定义。
- `icon_library_services.category` 是兼容主分类字段，`category_keys` 是 JSON 多分类数组；列表第一项作为主分类。
- `subscriptions.billing_type` 区分 `recurring` 周期订阅与 `one_time` 一次性买断。
- `subscriptions.is_keepalive` 保号套餐标记（短信保号），面向电信运营商分类的 `recurring` 订阅；表单入口只在电信运营商分类显示。该字段只控制提醒 / 续费弹窗 / 卡片等文案切换，不改变计费、续费推进、报表统计逻辑。
- `subscriptions.remind_days_before` 保存每个订阅自己的提前提醒天数（如 `7,1`），当前没有独立 `reminders` 表。
- `bundles` 用于套餐包 / 组合订阅管理，订阅通过 `bundle_id` 归属。
- `family_members` 是 JSON 数组，用于家庭共享成员展示。
- `notification_log.channel` 区分 `telegram` 与 `bark`。
- `activity_log` 支撑网页实时日志；普通用户看自己的记录，管理员可看全站记录。

## 6. 核心功能模块

1. **鉴权与用户管理**
   - 注册 / 登录 / Refresh Token。
   - SMTP 邮箱验证码（可选）。
   - 管理员审核、启停账号、管理员权限分配、密码重置、删除用户。
   - 管理员数据诊断：检查通知配置缺口、订阅脏数据、引用越权、保号范围异常、一次性买断残留周期字段、孤儿通知日志和近期失败通知。诊断项 code 包括：`telegram_config_incomplete`、`bark_config_incomplete`、`disabled_user_has_active_subscriptions`、`invalid_billing_type`、`invalid_cycle`、`invalid_cycle_count`、`negative_amount`、`currency_missing`、`category_missing` / `category_not_owned`、`payment_method_missing` / `payment_method_not_owned`、`bundle_missing` / `bundle_not_owned`、`subscription_missing_next_renewal`、`invalid_remind_days`、`keepalive_scope_invalid`、`one_time_has_recurring_fields`、`notification_user_missing`、`notification_subscription_missing`、`recent_notification_failures`。其中「引用缺失 / 越权、保号越界、一次性买断残留、无效提醒天数、周期订阅缺续费日」支持一键修复（`POST /api/admin/diagnostics/repair`，仅管理员）：修复前重判该问题仍存在、修复后复核引用合法性，需人工判断的项仍只报告。
   - 用户数据按 `user_id` 隔离。

2. **订阅管理**
   - 周期订阅 / 一次性买断。
   - 支持套餐名、备注、URL、VPS IPv4 / IPv6、家庭成员、套餐包、日历显示开关、同分类排序。
   - 续费日期计算：创建周期订阅未指定 `next_renewal_date` 时，从 `start_date` 推进到「今天及之后」的第一个到期日；`cycle_count` 小于 1 按 1 处理；非法 `cycle` 默认按月；月末日期会夹到目标月最后一天。
   - 支持续费后按当前日期重新计算，也支持按原到期日滚动。
   - 支持每订阅自定义 `remind_days_before`。
   - 删除订阅需提交当前用户密码校验。

3. **仪表盘、日历与报表**
   - 月度 / 年度支出、分类排行、最近付款、即将续费、已过期、永久购买统计。
   - 日历按 `next_renewal_date` 展示订阅，受 `show_in_calendar` 控制。
   - 日期字符串按本地日期解析，避免 `YYYY-MM-DD` 被 UTC 偏移。

4. **分类、付款方式与套餐包**
   - 系统预置 + 用户自定义。
   - 支持分类排序与订阅同分类拖拽排序。
   - 套餐包用于把多个订阅归组展示。

5. **多货币**
   - 预置主流货币，支持用户自定义货币。
   - 汇率缓存后按用户 `base_currency` 统计。

6. **通知提醒**
   - APScheduler 每天按 `REMINDER_SCAN_TIME` 扫描。
   - Telegram 和 Bark 可同时启用；任一通道失败不会阻断另一个通道。
   - 每次发送写入 `notification_log`，用于通知中心展示。
   - 管理员可运行提醒 dry-run 模拟：按指定日期预览哪些订阅会触发 Telegram / Bark、哪些会被跳过及原因；模拟只构造文案，不真实外发、不写通知日志。

7. **内置服务管理与图标系统**
   - 内置服务库支持多分类、启停、软删除 / 恢复、排序与图标预热。
   - 前端统一使用 `ServiceIcon` 渲染订阅与服务图标。
   - 支持 Emoji、上传图片、外部 URL、内置服务 favicon 与 fallback。

8. **备份恢复**
   - 普通用户可导出 / 导入自己的 JSON 备份。
   - 管理员可导出 / 恢复整站数据。

9. **实时日志与容器日志**
   - `activity_log` 提供网页实时日志。
   - 后端 stdout 记录请求概要、异常与慢请求，方便 Docker / NAS 排障。

## 7. 提醒调度逻辑

```text
每天 REMINDER_SCAN_TIME 定时任务:
  for 每个 active 且 billing_type=recurring 且有 next_renewal_date 的订阅:
    user = 订阅所属用户
    if user 不存在或 user.is_active=False: 跳过
    读取 subscriptions.remind_days_before，例如 "7,1"（去重）
    if next_renewal_date - today 命中任一提前天数:
      组装提醒文案
      if 用户启用 Telegram 且同日同通道未发过:
        发送 Telegram
        写 notification_log(channel="telegram", status="sent/failed")
      if 用户启用 Bark 且同日同通道未发过:
        发送 Bark（包含可选 bark_ttl）
        写 notification_log(channel="bark", status="sent/failed")
```

补充规则：

- 一次性买断订阅不参与周期到期提醒。
- 禁用用户（`is_active=False`）不参与提醒扫描，即使其订阅仍 active 且通道配置完整。
- 同一订阅同一通道同一天只发一次：同日去重以本地日历日（`TZ`）为准，仅匹配 `status="sent"` 的历史记录；失败记录不阻止当日重试。同一次扫描内还用内存集合兜底去重，避免未提交日志导致的重复发送。
- Telegram / Bark 互不依赖，失败独立记录。
- 用户可在网页里手动发送测试推送；管理员可触发提醒扫描用于排查配置（手动触发的扫描为全站扫描，仅管理员，普通用户返回 403）。
- 扫描并发互斥：`run_reminder_scan` 用非阻塞进程锁串行化，定时与手动并发或手动重复触发时，第二个直接跳过（返回 `skipped`，手动入口为 409），避免未提交日志互相不可见导致重复外发。该锁为进程级，覆盖当前单 worker uvicorn 部署；若改用多 worker（如 `gunicorn -w N`），每个进程独立 scheduler 与锁，需改用 DB 租约才能跨进程互斥。

## 8. 图标显示逻辑

前端统一使用 `ServiceIcon`：

- `src` 是图片 URL 时优先显示图片。
- 图片加载失败时切到 fallback。
- 旧透明 1x1 占位图会通过 `naturalWidth <= 1 && naturalHeight <= 1` 识别并切到 fallback。
- fallback 优先使用 Emoji，否则按服务名 / 订阅名首字符生成稳定颜色的字母图标。

后端图标库路径保持 `/api/icons/library/{slug}.png` 兼容，但响应的真实 `Content-Type` 可以是 PNG / ICO / WEBP / JPEG / SVG。远端 SVG 只有通过消毒后才会缓存；失败时返回可见 SVG fallback。

## 9. 部署方案

### 9.1 Docker Hub / GHCR 镜像部署

```bash
docker run -d --name subly \
  -p 8842:8000 \
  -e JWT_SECRET="$(openssl rand -hex 32)" \
  -e ADMIN_USERNAME=admin \
  -e ADMIN_PASSWORD=admin123 \
  -e ADMIN_EMAIL=admin@example.com \
  -e TZ=Asia/Shanghai \
  -v subly_data:/app/data \
  --restart unless-stopped \
  <你的用户名>/subly:latest
```

访问：`http://<host>:8842`；API 文档：`http://<host>:8842/docs`。

### 9.2 镜像 compose 部署

```bash
docker compose -f docker-compose.hub.yml up -d
```

`docker-compose.hub.yml` 默认映射 `8842:8000`，数据卷为 `subly_data:/app/data`。

### 9.3 源码 compose 部署

```bash
cp .env.example .env
# 编辑 .env 后：
docker compose up -d --build
```

`docker-compose.yml` 会构建本地镜像，并通过 Caddy 暴露 `80/443`。

### 9.4 镜像发布

GitHub Actions `docker-publish.yml` 支持：

- 推送到 `main` 或 `v*` tag 时自动构建并发布。
- 手动执行 `workflow_dispatch` 时填写 Docker 镜像版本号（如 `1.0.0`）。
- 同时发布 Docker Hub 与 GHCR 的 `latest` 和指定版本 tag。
- 版本号会按 Docker tag 规则校验。

## 10. 安全与隐私约束

- `JWT_SECRET` 必须使用随机强密钥。
- 管理员初始密码首次登录后应立即修改。
- `.env`、数据库文件、备份文件、图标缓存、上传文件等本地数据不应提交到仓库。
- 请求日志当前记录 method、path、status、duration、user_id、client 与 request_id，用于排障。
- 请求日志不得记录 request body、headers、query、密码、JWT、Bot Token、Device Key、API key、SMTP 密码、完整外部 URL、notes、remark、family_members 等敏感内容。
- 图标下载错误只记录必要的事件、slug、provider 和错误类型。
- 备份文件包含个人订阅与通知配置，导出后应妥善保管。
- 生产环境建议放在 HTTPS 反向代理后，并限制管理端入口暴露范围。
- CORS 当前为全开放（`allow_origins=["*"]` + `allow_credentials=True`），适合单服务自托管；如需多域名或公网暴露，建议在反代层收紧。
- 订阅 `url` 字段后端做协议白名单校验，仅允许 `http://` / `https://`，拒绝 `javascript:` / `data:` 等危险协议。
- 手动触发全站提醒扫描（`/api/notifications/run-scan`）仅管理员可用，普通用户返回 403；扫描为全站范围且会真实外发通知。
- Telegram / Bark 测试推送固定使用当前用户已存配置（Bot Token / Bark 服务器等），不接受请求 payload 覆盖出网目标与发送凭证，防止借后端做消息中继或 SSRF。
- 用户级出网目标（`telegram_api_base` / `telegram_proxy` / `bark_server`）经 `validate_outbound_url` 校验：仅允许 `http/https`，禁止 query / fragment / userinfo（防止把 `/bot.../getMe` 拼进 query 绕过约束），拦截链路本地（含云元数据 `169.254.169.254`）与危险协议；本机 / 私网不拦（自托管本地代理属正当用途）。启动迁移会清理历史不合规值。
- `/api/icons/from-url`（从任意 URL 下载图标，存在可读型 SSRF 风险）仅管理员可用；普通用户仍可上传图片、用内置服务库或直接填写图标 URL。
- 图标上传（`/api/icons/upload`）与 `from-url` 的 SVG 必须经白名单消毒（`_sanitize_svg_icon`，fail-closed），防止同源 SVG 存储型 XSS 窃取 `localStorage` 中的 token。
- 自定义服务 `domain` 写入前经 `_validate_icon_domain` 校验（接收原始输入，拒协议 / userinfo / 路径 / query / fragment，显式拒 `localhost`，字面 IP 与 DNS 解析后的地址都拦本机 / 私网 / 链路本地 / 元数据 / 非常规 IPv4 字面量）；favicon 与 `from-url` 抓取改为手动跟随重定向（`_safe_resolve`，`follow_redirects=False`），每一跳 host 经 `is_internal_host` 校验后才发请求，防「公网 302 跳内网」在复核前已发出。
- 备份导出只包含本人或系统级的分类 / 付款方式，不打包订阅越权引用到的他人私有元数据；备份导入在任何删除 / 写入前先做全量校验（缺 `subscriptions`、缺 name、非法日期、类型错误、辅助集合元素非对象均返回 400），整站恢复缺 `username` 的用户块报错而非静默跳过，避免「先清空再宽松导入」丢数据。
- 通知端点的外部调用失败统一返回泛化提示，底层异常细节（含 Bot Token 的请求 URL）只写入服务端日志，不回显给调用者，也不写入用户可见的 ActivityLog。
- 删除订阅需提交当前用户密码并通过校验，防止误删与他人操作。
- 订阅引用的分类 / 付款方式 / 套餐包做归属校验：仅允许系统级或本人账户下的引用，跨用户引用在创建 / 更新时被拒，历史脏数据由管理员诊断报告。

## 11. 当前演进方向

Subly 当前围绕“自托管续费雷达”持续收口：

- 将部署依赖聚焦到内置 SQLite，降低 NAS 和个人服务器维护成本。
- 保持中文单语言，减少后续开发维护负担。
- 提供 Telegram + Bark 双通道提醒，并支持 Bark 可选 TTL。
- 强化内置服务库、多分类、图标下载、SVG 消毒缓存与可见 fallback。
- 完善注册审核、用户管理、备份恢复、实时日志和文档一致性。
